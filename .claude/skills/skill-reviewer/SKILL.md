# Skill: Skill Reviewer

## 概要

Claude Code Skillのマークダウンファイルを厳格にレビューし、品質・構造・保守性の観点から問題点と改善提案を出力する。プロフェッショナルのコードレビューと同等の基準で評価する。

## トリガー

- 「Skillをレビューして」「スキルレビュー」「skill review」
- 特定Skillを指定: 「Skill Aをレビューして」「skill-c-myte-taxi-entryをチェック」
- 指定がない場合は `.claude/skills/` 配下の全Skillを対象にする

## レビュー観点

以下の観点で各Skillを厳格に評価する。問題がないものは「OK」とし、指摘がある場合は重要度を付ける。

### 1. 構造・可読性 (Structure)

- [ ] SKILL.mdの見出し構造は論理的か（H1→H2→H3の階層が正しいか）
- [ ] 各セクションの責務が明確か（1セクション1責務）
- [ ] 読み手が上から下に読むだけで処理フローを理解できるか
- [ ] 不要な重複記述がないか（DRY原則）
- [ ] 情報の配置場所は適切か（同じ情報が複数Skillに散在していないか）

### 2. 正確性・整合性 (Correctness)

- [ ] Skill間のデータフォーマット（JSON構造）は整合しているか
- [ ] セレクター/フィールド名が実際のデータファイルと一致しているか
- [ ] wbs_mapping.jsonの実際のスキーマとSkill Bの記述が一致しているか
- [ ] エントリJSONの実際のスキーマ（`data/pending/{id}.json`）とSkill Aの記述が一致しているか
- [ ] ファイルパス参照が実際のディレクトリ構成と一致しているか
- [ ] デフォルト値やEnum値（選択肢リスト）に矛盾がないか

### 3. 冗長性 (Redundancy)

- [ ] 複数Skillに同じ情報がコピペされていないか
- [ ] 共通化すべき定義（JSON構造、選択肢リスト、エラーハンドリング等）が各Skillに散在していないか
- [ ] 不要に長い説明やサンプルがないか
- [ ] 削除しても意味が変わらない記述がないか

### 4. 分割・統合の妥当性 (Modularity)

- [ ] 1つのSkillが大きすぎないか（目安: 300行超は分割検討）
- [ ] 1つのSkillの責務が多すぎないか（単一責任原則）
- [ ] 逆に、分割しすぎて文脈が分断されていないか
- [ ] 共通定義を別ファイルに切り出すべきか（共通スキーマ定義、共通セレクター等）
- [ ] Skill間の依存関係が明確に記述されているか

### 5. 保守性 (Maintainability)

- [ ] myTEのUI変更時に修正が必要な箇所が特定しやすいか
- [ ] セレクターがハードコードされている場合、変更時の影響範囲が明確か
- [ ] マジックナンバーや説明のない定数がないか
- [ ] バージョン管理や更新日時の管理ルールが明確か

### 6. 実用性・エッジケース (Robustness)

- [ ] エラーケースの記述は十分か
- [ ] ユーザーが想定外の入力をした場合の挙動が定義されているか
- [ ] タイムアウトやリトライの戦略が適切か
- [ ] 実際の運用で起こりうる問題（SSO切れ、ネットワーク遅延等）への対処が記述されているか

### 7. コンテキストウィンドウ効率 (Context Efficiency)

- [ ] Claude Codeのコンテキストウィンドウを無駄に消費する冗長な記述がないか
- [ ] 必要十分な情報量か（過不足なく）
- [ ] サンプルJSONが長すぎないか（必要最小限のフィールドで示せないか）
- [ ] 繰り返し参照される情報は外部ファイル参照に切り出すべきか

## レビュー手順

### Phase 1: 全Skill読み込み

1. `.claude/skills/` 配下の全ディレクトリを走査
2. 各 `SKILL.md` を読み込む
3. 関連データファイルも読み込む:
   - `.myte/wbs_mapping.json`（実際のスキーマ確認）
   - `data/pending/*.json` または `data/done/*.json`（実際のスキーマ確認）

### Phase 2: 個別レビュー

各Skillに対してレビュー観点のチェックリストを実行する。

### Phase 3: 横断レビュー

Skill間の整合性・重複・依存関係を横断的にチェックする。

### Phase 4: レポート出力

## 重要度の基準

| レベル | 基準 | 例 |
|---|---|---|
| 🔴 重大 | 動作不良・データ破損・Skill間連携の断絶 | JSONスキーマの不一致、存在しないフィールド参照、ファイルパスの誤り |
| 🟡 警告 | 保守性低下・冗長性・非効率 | 同じ情報の重複記述、300行超のSkill、曖昧な記述 |
| 🔵 提案 | 可読性・構造の改善 | セクション順序の最適化、サンプルの簡素化、命名の統一 |

## レビュー時の姿勢

- **甘い評価はしない**: 「問題なし」で済ませず、改善余地があれば必ず指摘する
- **具体的に指摘する**: 「冗長です」ではなく「L45-L60のJSON例はL120-L135と重複しており、共通定義への参照に置き換えるべき」のように行番号と修正案を示す
- **根拠を示す**: なぜ問題なのか、修正しないとどうなるかを明記する
- **実データと照合する**: Skillの記述と実際のデータファイル（wbs_mapping.json, data/pending/*/entry.json）を必ず突き合わせる
- **コンテキストコストを意識する**: Skillが読み込まれるたびにコンテキストを消費する。無駄な記述は実質的なコスト
